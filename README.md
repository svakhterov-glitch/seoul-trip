# TripsPlan — веб-планировщик путешествий

Пользователь регистрируется, создаёт поездку (страна/город + даты → каркас
календаря дней) и наполняет дни местами четырьмя способами:

- **✍️ Вручную** — ввод места и точка на карте;
- **🔗 По ссылке** — кинуть ссылку (Instagram, блог, Kakao/Google Maps) в инбокс,
  сервер разберёт её (координаты, фото, описание);
- **🔎 Поиском по названию** — ИИ + геокодер предлагают кандидатов в пределах города;
- **🤖 ИИ-сборкой маршрута** — живой веб-поиск по редакционным медиа и
  тревел-блогерам под город и **даты** поездки, с проверкой актуальности и
  **сезона**, кластеризацией по районам, часами и раскладкой по дням под темп
  (учитывает темы дней; есть добор мест в день).

Плюс: **перелёт и отели** (закреплённые плитки, отель ищется по названию и
ложится на карту), **замок места**, **чеклист** «что посмотреть/купить» с
ИИ-подсказками, **оптимизация порядка** точек в дне, блок **«Медиа»** (трендовые
места из подборок), **Telegram-предложка** (бот в группе складывает ссылки/фото/
пересланные посты → место в день или товар в **список покупок «🛍 Что купить»**),
и **слои-фильтры на карте** (наложить Медиа/Предложку поверх маршрута — видеть
кандидатов рядом). Готовая поездка «Сеул» — пример результата работы сервиса.
Подробное видение продукта — в [`PLAN.md`](PLAN.md); архитектура, соглашения и
«грабли» — в [`CLAUDE.md`](CLAUDE.md).

## Стек

- **Next.js 15 (App Router) + React 19 + TypeScript**, статический экспорт
  (`output: 'export'` → `out/`), без серверного рантайма.
- **Стили** — CSS Modules + глобальный `app/globals.css` (CSS-переменные). Шрифт — Manrope.
- **Хранение и вход** — **Supabase** (Auth + таблица `trips` с документом в JSONB, RLS).
- **Карта** — [Leaflet](https://leafletjs.com/) + тайлы CARTO Voyager.
- **Поиск/геокодинг** — **Kakao Local** (секрет `KAKAO_REST_KEY`), фолбэк Nominatim.
- **Серверная логика** — Supabase **Edge Functions** (Deno): `resolve-link`,
  `search-places`, `generate-itinerary`, `geocode`, `trends`, `suggest-checklist`,
  `telegram-webhook` (бот предложки).
- **Тесты** — Vitest + Testing Library (jsdom).

## Запуск

```bash
npm run dev      # дев-сервер → http://localhost:3000
npm run build    # статическая сборка в out/
npm test         # тесты (vitest run) — наш мини-CI
npm run lint     # next lint
```

Нужен `.env.local` с ключами Supabase (шаблон — `.env.local.example`). Без них
клиент Supabase бросит ошибку при первом вызове.

## Архитектура (коротко)

Слоистая, границы держим, чтобы менять хранилище и подключать ИИ без переписывания:

```
app/ (страницы, 'use client') → components/ (UI) → lib/ (модель + доступ к данным + сервисы)
                                                    Supabase (Auth + БД + Edge Functions)
```

- **`lib/entities.ts`** — чистая модель: типы (`TripDoc`, `Day`, `Place`…),
  фабрики и **иммутабельные мутации** (`addPlaceToTrip`, `movePlace`,
  `applyItinerary`…), без I/O.
- **`lib/trips.ts`** — единственный, кто знает форму строк таблицы `trips`.
- **Сеть/ИИ — за сервисами** (`resolveLink.ts`, `searchPlaces.ts`,
  `generateItinerary.ts`, `mediaBoard.ts`, `geocode.ts`, `suggestChecklist.ts`):
  UI зовёт их, а не `supabase.functions` напрямую. Edge-функции — в `supabase/functions/`.

Правила: UI не ходит в Supabase напрямую; мутации поездки иммутабельны и живут
в `entities.ts`; на каждую чистую функцию в `lib/` — тест рядом (`*.test.ts`).

## Развёртывание

Прод — VPS (домен **tripsplan.ru**, HTTPS, nginx отдаёт `out/`). **Пуш в `main` →
авто-деплой** (cron пуллит и пересобирает, если код изменился). Edge-функции
деплоятся отдельно: `SUPABASE_ACCESS_TOKEN=<sbp-токен> npx -y supabase functions
deploy <name> --project-ref <ref>` (Docker не нужен; токен отзывать после).

> Прежняя ванильная HTML/CSS/JS-версия лежит в `legacy/` и **не используется**.
