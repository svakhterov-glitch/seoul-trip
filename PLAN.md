# План развития · от лендинга к продукту-планировщику

Документ описывает, как из текущего статичного лендинга по Сеулу вырастить
продукт-планировщик путешествий. **Сейчас делаем только фронт** (без бэкенда),
но архитектуру закладываем правильную, чтобы Supabase и ИИ-генерация встали
на место без переписывания.

---

## 1. Целевой сценарий пользователя

```
1. Ввод перелёта и отеля
   (даты, рейсы туда/обратно, аэропорты, отель)
        │
        ▼
2. Сервис строит каркас календаря
   День 1 = прилёт (вечер) … Последний день = вылет (утро)
   дни пустые, ждут наполнения       ← сейчас это dayMeta/tabs
        │
        ▼
3. На каждый день / на всю поездку — три режима наполнения:

   ┌──────────────────────┬─────────────────────┬───────────────────┐
   │ 🤖 Авто через ИИ      │ ✍️ Собрать самому   │ 🔗 Кинуть ссылку  │
   │ ИИ + опыт путешест-в  │ ручной поиск/ввод   │ из Insta/блога/   │
   │ раскидывает по дням   │ места и времени     │ Kakao → в инбокс  │
   └──────────────────────┴─────────────────────┴───────────────────┘
        │
        ▼
4. Готовый маршрут: карта + расписание по часам   ← сейчас это всё уже есть
```

Текущая поездка в Сеул = **пример результата** работы сервиса. Наши 25 мест в
`data.js` становятся seed-базой «опыта путешественников» для режима 🤖.

---

## 2. Принципы архитектуры (закладываем сейчас, окупается потом)

Главная ошибка, которой избегаем: смешивать данные, хранилище и отрисовку.
Сейчас всё захардкожено в `data.js` + `app.js`. Разделяем на слои:

```
┌─────────────────────────────────────────────────────────┐
│  UI-компоненты (отрисовка, события)                       │
│  flightForm · calendar · dayView · placeCard · map ·      │
│  inbox · modeSwitch                                        │
└───────────────┬───────────────────────────────────────────┘
                │ читает состояние, вызывает действия
┌───────────────▼───────────────────────────────────────────┐
│  Store (состояние поездки в памяти + события подписки)     │
│  subscribe() / dispatch(action)                            │
└───────────────┬───────────────────────────────────────────┘
                │ persists через интерфейс
┌───────────────▼───────────────────────────────────────────┐
│  Repository (интерфейс хранилища)                          │
│  getTrip · saveTrip · addPlace · updatePlace · addLink …   │
│  ── LocalStorageRepository  (СЕЙЧАС)                       │
│  ── SupabaseRepository      (ПОТОМ — тот же интерфейс)     │
└───────────────┬───────────────────────────────────────────┘
                │
┌───────────────▼───────────────────────────────────────────┐
│  Model (сущности + фабрики + seed-данные Сеула)            │
└─────────────────────────────────────────────────────────────┘

  AI-сервис (отдельно): generateItinerary(trip, prefs)
  ── MockAiService  (СЕЙЧАС: отдаёт seed-места Сеула)
  ── ApiAiService   (ПОТОМ: вызывает серверную функцию + Claude API)
```

**3 правила, которые делают переход на бэкенд безболезненным:**

1. **UI никогда не трогает `localStorage` напрямую** — только через Repository.
   Замена на Supabase = новая реализация одного интерфейса.
2. **Все методы Repository асинхронны (`async`) уже сейчас**, хотя localStorage
   синхронный. Тогда переход на сетевые запросы не меняет сигнатуры.
3. **ИИ-генерация спрятана за интерфейсом `AiService`.** Сейчас mock возвращает
   готовые места Сеула; потом тот же вызов уходит на сервер.

## 3. Модель данных

```js
Trip {
  id, title, city, startDate, endDate,
  hotel: { name, coords:[lat,lng] },
  flights: [ Flight, Flight ]
}

Flight {
  direction: 'out' | 'back',
  airline, from, to,          // 'SVO Шереметьево', 'ICN Инчхон'
  date, depart, arrive,       // '2026-06-07', '17:30', '11:35'
  durationText, layoverText
}

Day {
  number,                     // 1..N
  date,                       // вычисляется из startDate
  cat,                        // 'start'|'tour'|'dist'|'shop'|'trend'|'final'
  title, sub,                 // заголовок и тема дня (редактируемые)
  mode                        // 'ai' | 'manual' | 'links' | null — как наполнялся
}

Place {
  id, tripId, dayNumber,
  name, coords:[lat,lng], time,
  photo, price, by, desc, history,
  source: 'seed' | 'manual' | 'ai' | 'link',
  sourceUrl?                  // если пришло из ссылки
}

InboxLink {                   // «не разобранные» ссылки
  id, url, status:'unsorted'|'placed',
  parsedName, parsedCoords, parsedPhoto,
  placedAsPlaceId?
}
```

## 5. Этапы реализации

### Этап 0 — Рефакторинг под слои (фундамент)
- Разнести `data.js`/`app.js` по структуре выше.
- Завести Model, Repository (localStorage), Store.
- Перенести нынешние 25 мест в `seed.seoul.js`.
- **Результат:** внешне сайт работает как сейчас, но данные идут через слои.
  Никакой `localStorage` в UI. Все методы Repository — `async`.

> **Решение по стартовому экрану:** при открытии сайт сразу показывает готовую
> поездку в Сеул (как сейчас). Создание новой поездки — отдельной кнопкой.

### Этап 1 — Экран ввода перелёта → каркас календаря
- Форма: город, даты, рейсы туда/обратно, отель.
- `days.js` генерит пустые дни между прилётом и вылетом.
- Текущий рендер дней/карты переиспользуется на сгенерированном каркасе.
- Перелёт из hero перестаёт быть захардкоженным — берётся из Trip.
- **Результат:** можно создать пустую поездку с нуля через интерфейс.

### Этап 2 — Режим ✍️ «Собрать самому» + drag-and-drop
- На дне кнопка «добавить место»: название, координаты (клик по карте), время.
- Редактирование/удаление места из интерфейса (не из кода).
- **Drag-and-drop:** перетаскивание карточек мест между днями и смена порядка
  внутри дня (опирается на `store.movePlaceToDay`, заложенный на Этапе 0).
- **Результат:** маршрут наполняется руками без правки файлов.

### Этап 3 — Режим 🔗 «Кинуть ссылку» (окно ссылок)
- Поле вставки URL → `InboxLink` со статусом `unsorted`.
- Лёгкий парсинг на фронте: координаты из ссылок Google/Kakao Maps, заголовок.
- Список «не разобрано» → **перетащить (drag-and-drop)** в нужный день → станет `Place`.
- **Результат:** твоё «окно для ссылок» работает целиком на фронте.

### Этап 4 — Режим 🤖 «Авто через ИИ» (заглушка)
- На дне/поездке кнопка «собрать маршрут».
- `AiService.generateItinerary()` → пока `MockAiService` отдаёт места Сеула
  из seed, раскидывая по дням по географии и времени.
- UI уже готов к настоящему ИИ — меняется только реализация сервиса.
- **Результат:** полный сценарий продукта работает на фронте end-to-end.

## 8. Блок «Медиа» — спека (трендовые места из редакционных материалов)

Будущий блок (вкладка с заголовком **«Медиа»**): показывать **трендовые места
города** — модные кафе/рестораны/смотровые и т.п. — взятые из редакционных
подборок, и выводить их **на карту** цветными метками. Заголовок «Медиа» — потому
что источник именно медиа-подборки. Спроектировано в обсуждении; кодом ещё не
реализовано. Ложится на заготовку edge-функции `trends`, категорию дня `trend` и
`PLACE_KINDS` (сегменты уже есть).

### 8.1 Решение по источнику: только редакционные рубрики

Из всех вариантов (соцсети, рейтинги Google/Foursquare, ИИ-как-редактор, медиа)
выбран **один путь — редакционные материалы**. Причина: рубрики «лучшее месяца /
новые открытия» дают сразу и **вкус** (отбор редакции), и **свежесть** (выходят
регулярно), и **явно называют места** + дают готовый абзац «чем интересно» —
ИИ такое разбирает надёжнее шумных соцсетей. Соцсети/Kakao/Naver-buzz — **не
берём** (закрытые API либо требуют синтеза).

Целевые рубрики (примеры):
- Англоязычные: **Time Out Seoul** («best new restaurants», «what's new»),
  **Visit Seoul** (тематические подборки), **Korea Herald/JoongAng** (лайфстайл).
- Корейские: **DiningCode** (트렌드 랭킹), **CatchTable**, новинки в Naver Place.

Доступ: где есть **RSS** — через него; иначе аккуратный fetch публичной
страницы. Контент не присваиваем — **показываем атрибуцию и ссылку на источник**
(«по версии Time Out Seoul, март 2026»). Пейволл-материалы пропускаем.

### 8.2 Свежесть (параметры, настраиваемые)

«Новые открытия» без контроля свежести через год станут враньём — поэтому:

| Параметр | Значение по умолчанию |
|---|---|
| Пересборка доски (cron) | раз в неделю |
| Окно публикации: рубрика «Новые» | статьи не старше 2 месяцев |
| Окно публикации: «В тренде месяца» | не старше 3 месяцев |
| Окно публикации: «Лучшие» | до 12 месяцев (свежие — выше) |
| Срок жизни карточки «Новые» | снять через 90 дней без нового упоминания |
| Срок жизни карточки «Лучшие» | до 12 месяцев |

На карточке всегда видны **дата публикации и источник**; внутри рубрики
сортировка — по свежести.

### 8.3 Поток данных

```
Реестр источников города (URL рубрик/RSS + тип рубрики)
        │  cron раз в неделю
        ▼
Забрать свежие статьи (RSS / fetch публичной страницы)
        ▼
ИИ извлекает: { name, blurb, segment, rubric, sourceDate, sourceUrl }
        ▼
Геокодер (Kakao/Google) → точные coords; что не нашлось — отбрасываем
        ▼
Фильтр свежести (8.2) + дедуп (одно место из разных медиа → слить, источники собрать)
        ▼
Кэш «трендовой доски» на город (TrendBoard)
        │  запрос пользователя
        ▼
Читаем кэш → отрисовка слоя на карте (8.5). Без живых запросов на клик.
```

### 8.4 Модель (добавится к entities)

```ts
TrendPlace {
  id, city,
  name, coords:[lat,lng],
  segment,      // ключ PLACE_KINDS: food/museum/nature/sight/shop/fun/bar/…
  rubric,       // 'new' | 'best' | 'trending'
  blurb,        // редакторская выжимка (1–2 предложения)
  source,       // 'Time Out Seoul'
  sourceUrl,    // ссылка на статью
  sourceDate,   // ISO-дата публикации — для свежести
  collectedAt   // когда собрали в доску
}
TrendBoard { city, updatedAt, items: TrendPlace[] }   // кэш на город
```

Добавление трендового места в поездку переиспользует поток инбокса
(`addInboxPlace` → перенос в день как `Place` с `source` и `sourceUrl`).

### 8.5 Навигация: «Медиа» — отдельная вкладка

«Медиа» — **самостоятельная вкладка, отдельная от маршрута**, стоит **слева от
кнопки «Весь маршрут»** в ленте вкладок (`DayTabs`):

```
[ Медиа ] [ Весь маршрут ] [ День 1 ] [ День 2 ] … [ Вылет ]
```

Реализация: добавить в `DayTabs` вкладку-режим слева от «Весь маршрут» (текущий
`activeDay`: 0 = весь маршрут, 1..N = дни; для «Медиа» — отдельное значение
режима, напр. `-1`/`'media'`). При выборе «Медиа»:
- карта показывает **слой трендовых меток** (8.6) вместо мест дня;
- **под картой — другая вёрстка** (8.7), не таймлайн дня.

### 8.6 Отрисовка на карте

Решение по меткам: **цвет = рубрика, эмодзи = сегмент** (одна переменная —
цветом, вторая — иконкой; две переменные цветом не кодируем — нечитаемо).

- 🟢 Новые · 🟡 Лучшие · 🔴 В тренде — легенда крошечная (всего 3 цвета).
  Сегмент — эмодзи из `PLACE_KINDS` (🍽 🏛 🛍 🌳 …).
- **Доступность:** красно-зелёная пара опасна для дальтоников → палитру сместить
  в безопасную (Новые = бирюзовый/синий, В тренде = оранжево-красный, Лучшие =
  золотой) **и дублировать рубрику текстом** в попапе/легенде (цвет — не
  единственный носитель смысла). Это требование раздела «Доступность» в CLAUDE.md.
- **Отдельный слой** поверх карты, визуально отличный от запланированных мест
  (напр. капля с пунктирной обводкой), чтобы подсказки не путались с местами в днях.
- **Фильтры-переключатели**: по цвету (рубрика) и по эмодзи (сегмент), вкл/выкл.
- При множестве меток — **кластеризация**. По клику — попап с blurb, источником+датой
  и кнопкой «добавить в инбокс/день».

### 8.7 Вёрстка блока под картой (медиа-стиль, отличается от дня)

Под картой в режиме «Медиа» — **не таймлайн дня** (`Timeline`/`PlaceCard` с
временем и порядком), а **витрина из квадратных плиток в стиле медиа-подборки**,
потому что и данные другие (редакционные карточки, а не пункты расписания):

- **Квадратные плитки** (мозаика-грид), не строки расписания: фото из материала
  (`og:image`) на всю плитку, поверх — название + эмодзи сегмента, **цветной чип
  рубрики** (Новые/Лучшие/В тренде), редакторская выжимка (`blurb`).
- **Связь плитка ↔ карта:** наведение/клик на плитку **подсвечивает на карте
  метку места с этим геотегом** (и наоборот — клик по метке подсвечивает плитку).
  Места без координат на карте не отрисовываются, но в витрине могут остаться.
- **Атрибуция обязательна:** «по версии Time Out Seoul · март 2026» —
  кликабельная ссылка на статью-источник (это и доверие, и корректное
  использование контента).
- Группировка/фильтр по сегменту (🍽/🏛/🛍…) и рубрике — те же переключатели,
  что у карты (8.6).
- У плитки — кнопка **«добавить в день/инбокс»** (поток `addInboxPlace`).
- Никаких полей «время/порядок/by» — они для запланированных мест, не для подсказок.

То есть `Timeline` для «Медиа» **не переиспользуем** — отдельный компонент
витрины (напр. `MediaBoard` + `MediaTile`) рядом с `planner/`.

### 8.8 Что понадобится при реализации

- Ключ геокодера для координат (Kakao Local или Google Places) — общий с поиском
