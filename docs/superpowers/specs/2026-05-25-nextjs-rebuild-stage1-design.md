# Пересборка на Next.js — Этап 1: лендинг + авторизация

**Дата:** 2026-05-25
**Статус:** утверждён к реализации
**Продукт:** TripsPlan (`tripsplan.ru`) — планировщик путешествий

## Контекст

Текущий сайт — ванильный HTML/CSS/JS (ES-модули, без сборки), развёрнут на VPS
`194.87.243.143` за nginx, автообновление из GitHub по cron. Бэкенд на Supabase
уже заложен: таблица `trips` (JSONB) с привязкой к пользователю и RLS-политиками
(`auth.uid() = user_id`), `SupabaseRepository`, edge-функции. Чего нет — экрана
входа во фронтенде.

Решение: пересобрать продукт на **Next.js + React + TypeScript**. Полная цель —
лендинг + вход + весь планировщик. Работа разбита на 3 последовательных этапа,
каждый оставляет работающий сайт. **Этот документ описывает только Этап 1.**

### Декомпозиция (для контекста, проектируется отдельно)
- **Этап 1 (этот спек):** каркас Next.js, лендинг, авторизация, экран «Мои поездки».
- **Этап 2:** планировщик (календарь дней, расписание, карточки мест, форма места, карта Leaflet) на данных Supabase.
- **Этап 3:** drag-and-drop, «окно ссылок» (resolve-link), ИИ-генерация, настройки поездки, форма новой поездки.

## Цель Этапа 1

Рабочий многопользовательский вход: посетитель видит лендинг с описанием
продукта, регистрируется/входит по email+паролю, попадает на экран «Мои поездки»
со своими данными из Supabase. Деплой сохраняет привычный поток «пуш в `main` →
сайт обновился».

Вне рамок: сам планировщик, создание/редактирование поездки, карта, ИИ, ссылки —
всё это Этапы 2–3. Открытие поездки и «Новая поездка» на Этапе 1 ведут на заглушку.

## Технические решения

- **Next.js (App Router) + React + TypeScript.**
- **Статическая сборка** (`output: 'export'`): лендинг пререндерится (быстро,
  хорошо для поиска); приложение за входом рендерится на клиенте. Это сохраняет
  «тупой» nginx, который отдаёт статику. Серверный рантайм Node не требуется в
  проде — только на этапе сборки.
- **Стили:** глобальные CSS-переменные (переносим палитру из `css/styles.css`:
  `--navy:#0f1b3d`, `--navy-2:#1c2c5e`, `--accent:#e8463c` и т.д.) + CSS-модули
  на компонент. Без тяжёлых UI-библиотек.
- **Supabase:** клиент `@supabase/supabase-js`, инициализация из публичных ключей
  (URL + anon/publishable key). Ключи кладём в переменные окружения сборки
  (`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`). Они безопасны для
  браузера (защищены RLS), как и сейчас в `config.local.js`.
- **Карта Leaflet** — только клиент (актуально для Этапа 2; учитываем при выборе
  статической сборки: динамический импорт с `ssr:false`).

## Структура репозитория

- Новый Next.js-проект — в корне репозитория.
- Текущий ванильный сайт переезжает в `legacy/` целиком (index.html, css/, js/,
  config.local.js остаётся вне git). Не удаляется: продолжает работать на старом
  сервере до переключения домена.
- `supabase/`, `docs/`, `deploy/` остаются в корне.
- `package.json` в корне заменяется на Next.js (скрипты `dev`/`build`/`start`/`lint`).

## Маршруты

- `/` — лендинг. Публичный, пререндер. Вариант компоновки **B**: слева описание
  продукта, справа форма входа.
- `/app` — «Мои поездки». Защищённый: при отсутствии сессии редирект на `/`.
  Так как сборка статическая, защита — клиентская: на маунте проверяем сессию
  Supabase, при отсутствии — `router.replace('/')`.

- **Шапка:** логотип «TripsPlan» (Plan — акцентным цветом), ссылка «Как это
  работает» (якорь), ссылка «Войти» (открывает форму во вкладке «Вход»).
- **Hero (две колонки):**
  - Заголовок: «Маршрут путешествия — по дням и по часам».
  - Лид: «Введите перелёт и даты — TripsPlan построит календарь поездки.
    Наполняйте дни местами: вручную, по ссылкам из блогов и карт или с помощью
    ИИ. Всё на одной карте.»
  - Пилюли: «Бесплатно», «Синхронизация между устройствами», «Карта и расписание».
  - Справа — `AuthCard`.
- **AuthCard:** переключатель «Регистрация / Вход», поля Email и Пароль, кнопка

- **Блок «Как это работает»:** три карточки — ✍️ «Соберите сам», 🔗 «Киньте

- Доступность: поля с label, фокус с клавиатуры, `prefers-reduced-motion`,

## Авторизация (Supabase, email + пароль)

- **Регистрация:** `supabase.auth.signUp({ email, password })`. Подтверждение
  почты в проекте Supabase **выключаем** (instant access) — включим позже, когда
  будет настроена отправка писем. После успеха — переход на `/app`.
- **Вход:** `supabase.auth.signInWithPassword({ email, password })` → `/app`.
- **Сессия:** Supabase JS персистит сессию в браузере; на загрузке приложения
  проверяем `getSession()`. Подписка на `onAuthStateChange` для реакции на
  выход/истечение.
- **Выход:** `supabase.auth.signOut()` в шапке приложения → возврат на `/`.
- **Ошибки** (показываем под формой, по-русски):
  - неверные учётные данные → «Неверный email или пароль»;
  - email занят → «Этот email уже зарегистрирован — войдите»;
  - слабый/короткий пароль → «Пароль должен быть не короче 6 символов»;
  - сетевая/прочая → «Что-то пошло не так. Попробуйте ещё раз».
- Состояние загрузки: кнопка с индикатором, поля блокируются на время запроса.

## Экран «Мои поездки» (`/app`)

- **Шапка приложения:** логотип, email текущего пользователя, кнопка «Выйти».
- **Контент:** сетка карточек поездок пользователя, прочитанных из `trips`
  (`select * where user_id = auth.uid()` — обеспечивается RLS, фронт просто
  делает `select`). На карточке: название, город, даты.
- **Пустое состояние:** «Пока нет поездок. Создайте первую» + кнопка «Новая
  поездка».
- **Заглушки Этапа 1:** клик по карточке и «Новая поездка» ведут на заглушку
  «Скоро» (полноценный планировщик и создание — Этап 2).

## Деплой

- На VPS ставится Node LTS (один раз).
- Шаг автообновления (`deploy/update.sh`) расширяется: `git fetch/reset` →
  `npm ci` → `npm run build` (Next.js static export в `out/`). nginx `root`
  переключается на `out/`. cron-механизм и «пуш в `main` → обновление»
  сохраняются; добавляется flock/лог, чтобы сборки не наслаивались (уже есть
  flock в cron).
- Переменные окружения сборки (`NEXT_PUBLIC_SUPABASE_*`) задаются на сервере
  (файл `.env.production`, вне git) — аналог нынешнего `config.local.js`.
- **Переключение домена:** пока Этап 1 не готов и не проверен, `tripsplan.ru`
  указывает на старый сайт. После проверки нового — деплоим на сервер и
  переключаем (или новый сервер становится основным).
- Старый ванильный сайт сохранён в `legacy/` и на старом сервере — откат
  возможен.

## Критерии готовности Этапа 1

1. На `/` открывается лендинг (вариант B) с фирменным стилем и финальными
   текстами; адаптивен; доступен с клавиатуры.
2. Можно зарегистрироваться по email+паролю и сразу попасть на `/app`.
3. Можно выйти и снова войти; сессия переживает перезагрузку страницы.
4. `/app` недоступен без входа (редирект на `/`).
5. На `/app` отображается email пользователя и его поездки из Supabase (или
   корректное пустое состояние).
6. Ошибки авторизации показываются понятными русскими сообщениями.
7. Проект собирается (`npm run build`) в статику и отдаётся nginx; деплой по
   пушу в `main` работает.

## Открытые вопросы / на потом

- Подтверждение email — включить на Этапе 2–3, когда будет настроена почта.
- Восстановление пароля («забыли пароль») — не входит в Этап 1.
- Перенос/seed поездки «Сеул» в аккаунт пользователя — решим на Этапе 2.
