# TripsPlan — Этап 2 (Next.js): экран открытия поездки (MVP) — дизайн

**Дата:** 2026-05-25
**Статус:** одобрен пользователем, готов к написанию плана
**Предыдущий этап:** `2026-05-25-nextjs-rebuild-stage1-design.md` (лендинг, авторизация, «Мои поездки», создание поездки)

## Цель

Дать возможность **открыть поездку** и увидеть её как планировщик: календарь дней,
карта с маркерами и маршрутом, наполнение дней местами (добавить / изменить /
удалить). Это перенос функциональности легаси-планировщика (`legacy/js/ui/*`) на
стек Next.js + React с привязкой к данным поездки в Supabase.

## Объём (согласовано)

**Входит в MVP:**
- Открытие поездки по ссылке-параметру.
- Автогенерация каркаса дней из дат поездки.
- Вкладки дней: «Весь маршрут» + День 1…N.
- Расписание дня: карточки мест (время, описание, ценовой бейдж, фото) с порядком.
- Карта Leaflet: маркеры мест, линия маршрута выбранного дня, выбор точки кликом.
- Добавление / редактирование / удаление места; сохранение в Supabase.

**Сознательно НЕ входит (на следующие итерации):**
- Drag-and-drop карточек мест.
- Настройка дня (смена категории/заголовка/времени дня) — дни идут с авто-заголовками.
- Окно ссылок (Этап «🔗»), ИИ-генерация (Этап «🤖»).
- Перелёты, отель как сущности, участники, бюджет, обложка.

## Поля места (согласовано)

- **Обязательные:** `name` (название), `coords` (точка на карте — выбор кликом).
- **Опциональные:** `time` (ЧЧ:ММ), `desc` (описание/заметка), `price`
  (`free`|1|2|3 → бесплатно/₽/₽₽/₽₽₽), `image` (URL фото).
- Категория — атрибут **дня** (как в легаси), не места. В MVP дни получают авто-категории
  (`start` у первого, `final` у последнего), редактор дня отсутствует.

## Маршрутизация и загрузка

Сайт — статический экспорт (`output: 'export'`), сервера нет. Динамический сегмент
`[id]` потребовал бы `generateStaticParams` со всеми id на этапе сборки — невозможно
для пользовательских данных. Поэтому:

- **Статическая страница `/app/trip/`**, id поездки — в query: `/app/trip/?id=<id>`.
- Клик по карточке в «Моих поездках» (`app/app/page.tsx`, проп `onOpen`) ведёт на
  `/app/trip/?id=<id>` вместо текущей заглушки `/app/new?soon=1`.
- Страница — клиентский компонент: проверяет сессию (`useSession`), читает `id`
  (`useSearchParams`), грузит поездку, рендерит планировщик. Нет сессии → `/`.
  Поездка не найдена/чужая (RLS) → понятное сообщение и ссылка на «Мои поездки».

## Модель данных

Расширяем `lib/entities.ts` (не ломая Этап 1):

- **`Place`**: `id`, `dayNumber` (1…N; `null` = не назначено), `order` (порядок в дне),
  `name`, `coords: [lat, lng] | null`, `time`, `desc`, `price: 'free'|1|2|3|null`,
  `image`, `photo` (эмодзи-заглушка `📍`), `source: 'manual'`.
- **`Day`**: `number`, `date` (подпись «8 июня»), `cat: string|null`, `title`, `sub`.
- **`createTripDoc`** дополняется: вызывает `buildDays(startDate, endDate)` → `days`
  заполнены при создании. Поле `places: []` остаётся.
- Старые поездки (созданные на Этапе 1 с `days: []`) при открытии догенерируют дни
  и сохраняются один раз.

Чистые помощники в `lib/entities.ts`: `placesForDay(trip, dayNumber)` (сортировка по
`order`, затем `time`), `getDay`, `lastDayNumber`, `normalizeDayOrders`,
`addPlaceToTrip`/`updatePlaceInTrip`/`removePlaceFromTrip` — **иммутабельные** функции,
возвращающие новый документ поездки (React-state-friendly; новое место — в конец дня).

## Новый день/дата-модуль

`lib/days.ts` — порт из `legacy/js/model/days.js` (с тестами):
`addDays`, `formatDateRu`, `formatDateRange`, `daysBetween`, `buildDays`.

## Слой данных Supabase

Дополняем `lib/trips.ts` (тот же контракт «клиент аргументом»):

- **`getTrip(supabase, id): Promise<TripSummary | null>`** — `select id,data` по id,
  `null` если строки нет (или RLS скрыл чужую).
- **`updateTrip(supabase, doc): Promise<TripDoc>`** — `update({ data: doc })` по
  `id = doc.id`. RLS-политика `trips_update` уже есть.

## Компоненты (`components/planner/`, тонкие)

- **`PlannerHeader`** — название, диапазон дат (`formatDateRange`), ссылка
  «← Мои поездки».
- **`DayTabs`** — вкладки «Весь маршрут» + дни; `role=tablist`, клавиатура, цвет
  вкладки = цвет категории дня. Порт `calendar.js`.
- **`Timeline`** — секции дней с карточками мест; кнопки «＋ Добавить место»,
  ✏️ изменить, 🗑 удалить; пустой день — подсказка. Порт `timeline.js` без
  DnD/«начало-конец дня»/истории/«кто нашёл»/Kakao.
- **`PlaceCard`** — одна карточка: фото/эмодзи, название, время, описание, бейджи
  (категория дня, цена).
- **`PlaceForm`** — панель добавления/редактирования: `name`, `time`, `desc`, `price`,
  `image` + «указать точку на карте» (включает режим выбора на карте). Валидация:
  `name` обязателен, `coords` обязателен (точка должна быть выбрана).
- **`TripMap`** — клиентский компонент Leaflet (грузится только в браузере), изолирует
  всю работу с картой: `update(trip, day)` рисует маркеры и линию маршрута,
  режим выбора точки кликом с черновым маркером. Порт `map.js`.
- **`helpers.ts`** — `catBadge`, `priceBadge` (порт, обобщённые — без Kakao/«кто нашёл»).

## Карта в Next.js (технические решения)

- Подключаем npm-пакет `leaflet` (+ `@types/leaflet`), CSS Leaflet импортируется в
  клиентском компоненте. Без `react-leaflet` — прямое управление, ближе к легаси и
  без риска несовместимости версий с React 19.
- `TripMap` грузится через `next/dynamic(() => …, { ssr: false })` — Leaflet требует
  `window`, а страница пререндерится при экспорте.
- Тайлы — OpenStreetMap (как в легаси).

## Поток данных

Без глобального стора. Страница `/app/trip/` держит поездку в `useState`. Мутации:
1. UI вызывает иммутабельный помощник (`addPlaceToTrip` и т.п.) → новый документ.
2. `updateTrip(supabase, doc)` сохраняет.
3. `setTrip(doc)` обновляет состояние → перерисовка таблайна и карты.

Активный день — локальный `useState` (0 = «весь маршрут»), не персистится.

## Обработка ошибок

- Поездка не найдена / нет доступа → сообщение + ссылка «Мои поездки».
- Ошибка сохранения места → неблокирующее уведомление, состояние не теряется.
- Загрузка → индикатор «Загрузка…».

## Тестирование (TDD, как на Этапе 1)

- **Юнит (Vitest):** `lib/days.ts` (границы дат, формат), `lib/entities.ts`
  (`buildDays` в `createTripDoc`, `placesForDay`, иммутабельные мутации), `lib/trips.ts`
  (`getTrip`/`updateTrip` с фейковым клиентом).
- **Компоненты (RTL):** `Timeline` (рисует места, кнопки вызывают колбэки),
  `PlaceForm` (валидация, submit с данными), `DayTabs` (переключение).
- `TripMap` (Leaflet) — без юнит-теста (DOM-карта), проверяется вручную/сборкой.

## Деплой

Изменений в инфраструктуре не требуется: пуш в `main` → cron на сервере пересобирает
(`deploy/update.sh`, сборка только при изменениях) и отдаёт `out/`. Новые маршруты:
`out/app/trip/index.html`.

## Критерии готовности

1. Клик по карточке поездки открывает `/app/trip/?id=…`.
2. Открытая поездка показывает дни (даже если создана пустой на Этапе 1).
3. Вкладки переключают день; «Весь маршрут» показывает всё.
4. Можно добавить место (название + точка на карте + опц. поля) — появляется в
   расписании и на карте, сохраняется после перезагрузки.
5. Можно изменить и удалить место; изменения сохраняются.
6. Карта рисует маркеры и линию маршрута дня; пустой день — без линии.
7. Сборка `npm run build` создаёт `out/app/trip/index.html`; все тесты зелёные.
8. HTTPS-сайт работает, регрессий на лендинге/входе/«Моих поездках» нет.
